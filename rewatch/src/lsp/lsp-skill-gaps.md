# LSP Skill Gaps: Dependency Usage Analysis

## Problem Statement

When an AI agent needs to answer "how much of a dependency do we use?" (e.g. which parts of `@rescript/webapi` are referenced in a project), the existing LSP skills cannot answer this. The practical fallback is `grep WebAPI\. src/**/*.res`, which defeats the purpose of having structured tooling.

## Current Skills (context)

Three Claude Code skills exist for ReScript projects (defined in `.claude/skills/`):

- **rescript-lookup**: HTTP endpoints on the running LSP (`/hover`, `/definition`, `/references`, `/document-symbols`, `/workspace-symbols`) at `http://127.0.0.1:<port>/`. Requires absolute file paths and 0-based line/col positions.
- **rescript-db**: SQLite database (`rescript.db`) generated by `rescript sync`. Contains tables: `packages`, `modules`, `types`, `fields`, `constructors`, `values`, `aliases`, `usages`. Indexes what packages *offer* and what projects *use*.
- **rescript-diagnostics**: HTTP endpoint (`/diagnostics`) that returns current build errors.

## Implemented

### Gap 3: usages table in rescript.db (IMPLEMENTED — needs polishing)

The `usages` table tracks cross-module references — which project modules reference which dependency symbols. The core feature works end-to-end: `rescript sync` populates usages, and the LSP keeps them incrementally updated via a background queue. Several rough edges remain.

**Schema** (in `rewatch/src/llm_index.rs`):

```sql
CREATE TABLE usages (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    source_module_id INTEGER NOT NULL,   -- project module that contains the usage
    target_module_id INTEGER NOT NULL,   -- dependency module being referenced
    target_path TEXT,                    -- dot-joined path within the module (e.g. "Storage.getItem")
    tip TEXT NOT NULL,                   -- symbol kind: Value, Type, Field(name), Constructor(name), Module
    source_line INTEGER,                 -- 0-based line
    source_col INTEGER,                  -- 0-based column
    FOREIGN KEY (source_module_id) REFERENCES modules(id) ON DELETE CASCADE,
    FOREIGN KEY (target_module_id) REFERENCES modules(id) ON DELETE CASCADE
);
```

**How it works**:

1. **OCaml side** (`analysis/src/LlmIndex.ml`): `extractUsages` reads `extra.externalReferences` from each `.cmt` file. Namespace references (e.g. `WebAPI` + `["DOMAPI", "element"]`) are resolved to the actual namespaced module name (`DOMAPI-WebAPI` + `["element"]`) via `resolveNamespacedTarget`. For modules with both `.res` and `.resi`, the structure (types, values) is extracted from the `.cmti` (public API) but usages are extracted from the `.cmt` (implementation), since the implementation is where actual dependency references live.

2. **Rust side** (`rewatch/src/llm_index.rs`): Two-pass insertion — all modules are inserted first, then `insert_usages` resolves `qualified_name` → `module_id` via a pre-built HashMap. Usages whose target module isn't in the DB are logged at `tracing::debug!` level and skipped (expected for internal/unindexed modules). The `modules` table stores a `cmt_hash` (blake3 of the `.cmt`/`.cmti` file) used by the incremental sync to skip unchanged modules.

3. **Incremental updates** (`rewatch/src/lsp/queue/db_sync.rs`): A background `DbSyncQueue` receives events after each LSP flush and after the initial build. It debounces/coalesces events, compares `cmt_hash` values against the DB to filter out unchanged modules, then shells out to the analysis binary for only the changed modules, deletes their old usages, and inserts new ones. Uses `OutputTarget::Lsp` paths since the LSP writes `.cmt`/`.cmti` to `lib/lsp/`.

4. **Initial sync** (`rewatch/src/lsp/queue.rs`): `trigger_initial_db_sync` fires a full sync (all modules) right after the queue is created, so the usages table is populated without waiting for the first file edit. Subsequent syncs use `cmt_hash` comparison to skip modules that haven't changed.

**Key files**:
- `analysis/src/LlmIndex.ml` — `extractUsages`, `resolveNamespacedTarget`, dual `.cmt`/`.cmti` loading
- `rewatch/src/llm_index.rs` — `AnalysisUsage`, `insert_usages`, `build_module_id_map`, schema DDL
- `rewatch/src/lsp/queue/db_sync.rs` — `DbSyncQueue`, `process_batch`, hash-based filtering, OTEL instrumentation
- `rewatch/src/lsp/queue.rs` — `build_sync_event`, `send_db_sync_events`, `trigger_initial_db_sync`
- `rewatch/src/lsp.rs` — `ensure_runtime_module_data`, initial sync wiring

**Known issues to polish**:

1. **`cmt_hash` UPDATE uses `name` instead of `qualified_name`**: In `db_sync.rs`, `UPDATE modules SET cmt_hash = ?1 WHERE name = ?2` uses the non-unique `name` column. For top-level modules this works because `name` happens to be unique, but nested modules (e.g. two different parents both containing a `Styles` submodule) could collide. The SELECT for hash comparison (`SELECT name, cmt_hash FROM modules`) has the same issue — the HashMap keeps only the last row per name. **Fix**: use `qualified_name` for both the SELECT and UPDATE. The `ModuleFileEntry` would need to carry `qualified_name` alongside `module_name`, or `build_sync_event` would need to resolve it.

2. **First edit after startup re-syncs most modules**: OTEL traces show that the first file edit after LSP startup triggers `hash_mismatch` for ~354/357 modules, even though the initial sync (batch 1) found all hashes matching. This happens because the initial build's `CompilationStage.cmt_hash` matches the DB, but the first incremental build re-typechecks the entire reverse-dependency closure (see `skip-redundant-dependent-typecheck.md`), producing new `.cmt` files with different content. The `cmt_hash` comparison correctly detects these as changed. **Mitigation**: the `.cmi`-based skip optimization described in `skip-redundant-dependent-typecheck.md` would reduce the reverse-dep closure to near-zero for body-only edits, making this a non-issue.

3. **No test coverage for incremental sync path**: The snapshot test (`llm-index.test.mjs.snap`) only covers the analysis binary's JSON output for a simple module. There are no tests for: the `db_sync` consumer, hash-based filtering, debounce/coalescing, or the initial sync trigger. These are dogfooded via OTEL traces for now.

## Remaining Gaps

### Gap 1: workspace-symbols doesn't find symbols from dependency packages

**Symptom**: Searching for `WebAPI` or `WebAPIX` via `/workspace-symbols?query=...` returns `[]`.

**Root cause** (in compiler repo): `rewatch/src/lsp/workspace_symbol.rs` only collects `.ast`/`.iast` files from the project's own build state modules. Dependency packages in `node_modules` are excluded from the file list passed to the analysis binary. The analysis binary's symbol walker (`analysis/src/WorkspaceSymbol.ml`) would handle dependency files fine — they're just never passed in.

**Fix**: Extend the module collection loop in `workspace_symbol.rs` to also iterate dependency package modules. Consider a performance guard (e.g. only include deps when query length > 3, or add an explicit `includeDependencies` flag).

**Effort**: Moderate. The Rust handler needs a broader iteration scope; the OCaml analysis code needs no changes.

### Gap 2: references doesn't cross package boundaries reliably

**Symptom**: Running `/references` on a `MouseEvent.stopPropagation` call in project code returns `[]` — it doesn't find other call sites in the same project.

**Root cause**: `analysis/src/References.ml`, function `allReferencesForLocItem` (around line 520-573), scopes its search to `package.projectFiles`. For symbols defined in dependencies (like `@rescript/webapi`), the function resolves the definition file but then only searches within project files for usages. The search walks outward from the definition, and cross-package resolution can fail for namespaced packages.

**Two sub-problems**:
1. Single-symbol cross-package references (e.g. "find all project usages of this specific `WebAPI` function") — likely fixable by adjusting package scoping in `References.ml`.
2. Bulk "find all usages of any export from package X" — now addressable via the `usages` table (Gap 3).

**Effort**: Sub-problem 1 is a targeted fix. Sub-problem 2 is solved by Gap 3.

### Gap 4: No "search references by qualified name" endpoint

**Symptom**: `/references` requires `file`, `line`, `col` — you must already know a position where the symbol is used. There's no way to say "find all references to `WebAPI.Storage.getItem`" by name alone.

**Root cause**: The HTTP server in `rewatch/src/lsp/http.rs` only exposes position-based endpoints. No name-based reference search exists.

**Implementation path**: Add `GET /usages?module=StorageAPI-WebAPI&name=getItem` as a thin wrapper around a `rescript.db` SQL query on the `usages` table. Trivial effort now that Gap 3 is implemented.

**Effort**: Low.
